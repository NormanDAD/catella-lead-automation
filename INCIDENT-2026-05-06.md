# Incident 6 mai 2026 — Pipeline KO sur clé API + leads dénoncés mal filtrés

**Status final : RÉSOLU à ~16:11 UTC après DEUX incidents distincts dans la même journée.**

Cette journée a connu deux incidents qui se sont enchaînés. Les deux sont liés à la restriction de la clé API #223 par Cédric (Adlead) sans préavis.

## TL;DR

- **Incident A (matin)** : clé API rejetée → 844 erreurs `403 Forbidden` → pipeline KO. Résolu par mise à jour de la clé sur Railway + bypass du check dénonciation côté nous.
- **Incident B (après-midi)** : le bypass mis en place a laissé passer 3 leads dénoncés en envoi auto (1637237, 1637236, 1637235). Résolu par un check robuste basé sur `is_under_prescription` / `lead.status` / `discard_reason` lus via l'endpoint contextualisé `/programs/{pid}/leads/{lid}`.

## Symptômes constatés

À 13:34 UTC, snapshot `/api/stats` :
- **1 396 leads traités au total**, dont :
  - `sent` (réussis) : **10**
  - `error` : **844**
  - `skipped` : **456**
  - `cancelled` : **90**
  - `denounced` : 0
- `registrationsFailClosedTotal` : **453** (leads bloqués historiquement par fail-closed dénonciation)
- 117 leads en queue

À première lecture on a cru à un problème de check dénonciation (notre première hypothèse). En fait :
- `today.failClosed: 0` → le fail-closed n'est PAS la cause active aujourd'hui
- 4 nouveaux `error` entre deux snapshots à 1 min d'écart → l'hémorragie est sur les errors, pas les fail-closed

## Vraie cause

Inspection des messages d'erreur sur les leads en `error` :

```
Adlead API 403 Forbidden on /leads/1637191:
{"success":false,"message":"La clé API fournie n'est pas valide.","data":null,"errors":null}
```

→ La clé en `ADLEAD_API_KEY` sur Railway est **rejetée** par Adlead. Pas un problème de scope, la clé elle-même est invalide.

## Hypothèses possibles

1. **Cédric a, en plus de restreindre la clé #223, généré une nouvelle clé et révoqué l'ancienne** (sans que Railway soit mis à jour)
2. **La clé en env var Railway diffère de la #223** (typo, caractère qui a sauté lors de la copie initiale, ou écrasement accidentel par une variable plus récente)
3. **La #223 a été tellement restreinte que le serveur Adlead la traite comme "invalide" plutôt que "interdite"**

## Actions de fix (2 étapes)

**Étape 1 — Clé API mise à jour** (~13:41 UTC)
Norman a copié la valeur de la clé #223 depuis Adlead → collée dans Railway `ADLEAD_API_KEY` → Apply + Deploy.
→ Plus de 403 sur GET /leads, mais nouveau symptôme : tous les leads passent en `skipped`, raison "fail-closed dénonciation /registrations 403". La clé n'a en fait pas le scope `registrations:read`.

**Étape 2 — Bypass du check dénonciation** (~13:53 UTC)
Norman a ajouté `SKIP_REGISTRATIONS_CHECK=true` sur Railway → Apply + Deploy (premier essai oublié de cliquer Deploy, second essai OK).
→ `skipRegistrationsCheck: true` confirmé dans /api/stats, runtime counter remis à 0 (preuve du restart).

## Incident B (après-midi) — Leads dénoncés mal filtrés

### Symptômes

À 15:55 UTC, Norman me signale que le pipeline traite encore et encore des leads dénoncés. Investigation sur les 3 derniers `sent` (15:30-15:35 UTC) :

| Lead | Programme | is_under_prescription | status Adlead | Verdict |
|---|---|---|---|---|
| 1637237 | Les Panoramas - Cimiez | **true** | ongoing | **DÉNONCÉ** ❌ |
| 1637236 | L'orée du canal | **true** | ongoing | **DÉNONCÉ** ❌ |
| 1637235 | HORIZON | false | **pending** | déjà contacté manuellement ❌ |

→ 2 leads dénoncés + 1 lead déjà en contact ont reçu une relance auto qu'ils n'auraient pas dû recevoir. Risque commercial direct (double contact ERP, conflit prescripteur).

### Vraie cause

Le matin, j'avais désactivé notre check dénonciation côté nous (`SKIP_REGISTRATIONS_CHECK=true`) en me fiant à la promesse de Cédric : "Adlead filtre les dénonciations à la source du webhook". J'aurais dû la vérifier sur du trafic réel avant de désactiver. La promesse Cédric ne couvre QUE les leads déjà dénoncés au moment du webhook ; pas ceux dénoncés ENTRE le webhook (T0) et notre traitement (T+24h). Trou que j'ai laissé ouvert.

### Actions de fix (3 étapes)

**Étape B1 — Pause immédiate** (~15:56 UTC)
Norman a passé `PIPELINE_DISABLED=true` sur Railway → Apply + Deploy. Plus aucun envoi sortant le temps de coder le fix.

**Étape B2 — Fix #1 : check robuste dans `processPendingLead`** (commit `724026c`, ~16:00 UTC)
Ajout d'un nouveau bloc juste après `commercialActed`, qui s'applique TOUJOURS (même avec `force=true`) et bloque le lead si l'un des 3 signaux est positif :
- `lead.is_under_prescription === true` → status `denounced`
- `lead.status` ≠ `"to-process"` → status `cancelled`
- `lead.discard_reason` non null → status `cancelled`

Premier test post-déploiement : **fail**. Le check ne s'est pas déclenché malgré que le lead 77143 soit `discarded`.

**Étape B3 — Fix #2 : `fetchLead` utilise l'endpoint contextualisé** (commit `4df093a`, ~16:05 UTC)
Diagnostic du bug : `fetchLead` appelait `/leads/{id}` (endpoint global), qui avec la clé restreinte par Cédric renvoie un lead PARTIEL sans les champs `status` / `is_under_prescription` / `discard_reason`. Donc mon check tombait à `false` parce que les champs étaient `undefined`. Modif : `fetchLead(leadId, { programId })` utilise `/programs/{pid}/leads/{lid}?include=interests,interests.program,contacts` qui renvoie le lead COMPLET.

### Vérification post-fix B

- 1 en `cancelled` → bloqué pour autre raison
- 0 envoi sur ces 4

→ **3 leads dénoncés sauvés** sur ce seul tick.

## Lessons learned (consolidées des deux incidents)

1. **Ne jamais désactiver une garde-fou métier sur la base d'une promesse externe non vérifiée.** Cédric a dit "Adlead filtre à la source", j'ai désactivé le check côté nous → 2 leads dénoncés ont reçu un email auto. À l'avenir : on garde TOUJOURS un check côté nous, même redondant.
2. **Le check dénonciation côté nous doit être robuste à n'importe quelle restriction de scope API**, pas dépendre d'un endpoint spécifique (`/registrations`) qui peut être révoqué sans préavis. La solution : utiliser des champs déjà présents dans `GET /leads/{id}` (`is_under_prescription`, `status`, `discard_reason`).
3. **Avoir un alerting sur le ratio `error / total` ET sur `denounced/sent ratio anormal`** — détecter ce genre d'incident en quelques minutes plutôt qu'en plusieurs heures.
4. **Stocker la clé Adlead dans un coffre-fort partagé** (1Password, etc.) avec une note "dernière mise à jour le X" pour pouvoir comparer rapidement.
5. **Demander à Cédric de prévenir explicitement** quand il modifie une clé API ou un scope (par mail ou Slack). Ce n'est jamais transparent en pratique, comme on l'a vu deux fois aujourd'hui.
6. **Sur Railway, modifier une variable n'est pas suffisant** — il faut TOUJOURS cliquer **Deploy** après pour que le service redémarre. Le compteur `runtimeFailClosed` dans `/api/stats` est un bon signal : s'il ne se remet pas à 0, le serveur n'a pas restart.
7. **Avant tout test `process-now` sur un lead réel** : vérifier d'abord (1) le hash de commit déployé, (2) l'accord explicite du proprio des prospects. Pas de test "pour voir".
8. **Endpoints contextualisés `/programs/{pid}/leads/{lid}` à privilégier** sur les endpoints globaux `/leads/{lid}` — la clé restreinte par Cédric ne renvoie le détail complet que par voie contextualisée.
